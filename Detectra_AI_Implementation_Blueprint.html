01

System Architecture

High-Level Architecture Overview

Detectra AI is structured as a 6-layer production architecture with clear boundaries between ingestion, processing, intelligence, reasoning, storage, and presentation. All layers communicate via async message queues to achieve non-blocking, scalable throughput.

Layer 1 · Ingestion

RTSP stream reader, video file uploader, frame extractor, audio splitter. Entry point for all video data.

Layer 2 · Preprocessing

Frame normalization, resizing, denoising, optical flow, audio resampling & spectrogram generation.

Layer 3 · Perception

Object detection, logo recognition, face/pose detection, action recognition, speech-to-text, environmental audio.

Layer 4 · Reasoning

Multimodal fusion engine (transformer), temporal event sequencing, context inference, story/scene understanding.

Layer 5 · Storage

PostgreSQL for metadata, Redis for live event stream, S3-compatible blob for video/thumbnails, TimescaleDB for time-series events.

Layer 6 · API + UI

FastAPI REST + WebSocket gateway, Next.js dashboard, real-time timeline, alert engine, report export.

Component Responsibilities

Component Responsibility Technology
StreamIngester Opens RTSP/HTTP streams via OpenCV or FFmpeg, extracts frames at configurable FPS (default 5fps for analysis, 25fps passthrough), buffers into async queue. OpenCV · FFmpeg · asyncio
FrameRouter Dispatches frames to parallel GPU workers. Load balances across available CUDA devices. Drops frames if queue saturated (priority: keyframes). Redis Streams · Celery
PerceptionHub Orchestrates parallel calls to 5 independent inference workers: Detector, Tracker, ActionNet, LogoNet, AudioNet. Each is an isolated microprocess. PyTorch · ONNX Runtime
FusionEngine Consumes per-frame perception outputs, aligns temporal timestamps, runs cross-modal transformer to produce fused scene embeddings per 2-second window. HuggingFace Transformers
ContextReasoner Sliding-window event buffer (30s). Detects anomaly patterns, generates natural-language scene descriptions, triggers alert conditions. LLaMA / Mistral fine-tuned
AlertAgent Rule engine + ML anomaly scorer. Fires webhooks, stores alert events, sends WebSocket push to dashboard in <500ms. FastAPI WebSocket · Celery Beat
TimelineDB Writes all timestamped events (detections, actions, audio, alerts) into TimescaleDB hypertables for fast range queries used by dashboard. TimescaleDB · SQLAlchemy
DashboardAPI FastAPI REST + WebSocket. Serves timeline queries, live event push, report generation, camera management, user auth (JWT). FastAPI · Pydantic v2

Data Flow (Step-by-Step)

RTSP / MP4 → StreamIngester → FrameQueue (Redis) → PerceptionHub → FusionEngine → ContextReasoner → TimelineDB + AlertAgent → Dashboard WebSocket
ℹ

Audio is parallel, not sequential. The audio pipeline branches from the ingester simultaneously with the visual pipeline. Both streams converge at FusionEngine with a shared timestamp index. This is critical for correct AV synchronization in the fusion model.

AI Pipeline Design

Complete Video Intelligence Pipeline

1
Video Input & Stream Handling
OpenCV/FFmpeg opens RTSP stream or file. Frames extracted at 5fps for inference (25fps for smooth playback stored separately). Audio extracted as 16kHz mono PCM. Concurrent streams handled via asyncio + thread pool. Frame timestamps embedded as metadata from presentation timestamp (PTS).
OpenCV VideoCapture FFmpeg subprocess asyncio.Queue
2
Preprocessing
Frames: resize to 640×640 (letterboxed), normalize to [0,1], CLAHE contrast enhancement for low-light. Optical flow (Farneback) computed between consecutive frames to provide motion priors. Audio: VAD (Voice Activity Detection) with Silero, spectrogram via Librosa (mel-spectrogram, 128 bins, 2048 FFT).
Albumentations Librosa Silero VAD CLAHE (cv2)
3
Object Detection + Logo Recognition
YOLOv8x for general object detection (80 COCO classes + custom person subclasses). RT-DETR as fallback for crowded scenes where YOLO NMS fails. Logo recognition uses a ViT-B/16 classification head fine-tuned on OpenLogos-32 + custom surveillance-specific logos. Confidence threshold: 0.45 for detection, 0.70 for logo. Non-Maximum Suppression via Torchvision.
YOLOv8x (Ultralytics) RT-DETR ViT-B/16 OpenLogos-32
4
Multi-Object Tracking
ByteTrack for robust ID assignment across frames (handles occlusion and re-ID). Each tracked person assigned a persistent integer ID for the session. Track states: Active / Lost / Removed. Re-identification uses appearance features from ReID model (OSNet) to recover lost tracks. Track history buffers last 150 bounding boxes per ID for trajectory analysis.
ByteTrack OSNet ReID Kalman Filter
5
Action Recognition
Two-stage: (a) Pose estimation via YOLOv8-Pose → keypoint sequences fed into PoseFormer for action classification over a 16-frame sliding window. (b) SlowFast network for coarse action categories (walking, running, fighting, loitering, gathering) over 8-frame clips. Actions bound to tracker IDs. Temporal fusion: actions carry a duration estimate (start_frame, end_frame, confidence). Custom fine-tuning on HMDB51 + NTU RGB+D for surveillance-relevant classes.
YOLOv8-Pose PoseFormer SlowFast (R50) HMDB51 NTU RGB+D
6
Audio Intelligence
Speech-to-text: OpenAI Whisper (medium model, runs locally via whisper.cpp for speed). Segments of 30s processed with 5s overlap. Environmental sound: PANNs (Pretrained Audio Neural Networks) on AudioSet-20K — classifies 527 sound events. Key classes for surveillance: glass breaking, gunshot, screaming, crowd noise, vehicle. Results timestamped and merged with video timeline.
Whisper Medium (local) PANNs / CNN14 AudioSet-20K whisper.cpp
7
Multimodal Fusion Engine
Temporal window of 2 seconds. Inputs: per-frame detection embeddings (YOLOv8 backbone features), action class logits, audio event probabilities, optical flow magnitude. A 4-head cross-attention transformer aligns modalities using synchronized timestamp as positional encoding. Output: a 512-dim fused scene embedding per window, plus a joint event descriptor JSON (objects present, actions occurring, sounds detected, confidence scores).
Cross-Attention Transformer HuggingFace BERT backbone Temporal Positional Encoding
8
Context Understanding & Story Inference
A sliding 30-second event buffer feeds a fine-tuned language model (Mistral-7B-Instruct or LLaMA-3-8B via llama.cpp). The LLM receives a structured prompt: {scene_description + timeline_of_events} → generates: (1) natural-language scene summary, (2) anomaly score [0–1], (3) intent classification (benign / suspicious / urgent), (4) recommended action. This is the "story layer" — it synthesizes what happened, not just what was detected.
Mistral-7B-Instruct (4-bit GGUF) llama.cpp Structured Prompt Engineering Anomaly Scoring
9
Output: Events, Alerts, Timeline
All timestamped events written to TimescaleDB. Alert conditions (anomaly_score > 0.75, specific actions detected) trigger the AlertAgent which pushes WebSocket messages to the dashboard in <500ms. Reports generated as JSON (structured), PDF (rendered), and CSV (tabular) on demand. Dashboard shows interactive timeline with event markers, bounding box replay, and LLM-generated summaries per segment.
TimescaleDB hypertable WebSocket push ReportLab PDF Chart.js timeline

Multi-Agent Design

Detectra AI uses a 4-agent architecture where each agent is a specialized, independently scalable process. Agents communicate through Redis Streams. A Coordinator agent manages the hierarchy and resolves conflicts between agent outputs.

StreamAgent
Ingests and preprocesses raw video/audio streams. The gatekeeper — nothing enters the pipeline without passing through this agent.

Responsibilities: RTSP connection management, frame extraction at variable FPS, audio splitting, CLAHE preprocessing, optical flow computation, timestamp embedding, frame routing to Redis Stream.

INPUTS
RTSP stream URL MP4/AVI file Camera config JSON
OUTPUTS
FramePackets → Redis AudioChunks → Redis
PerceptionAgent
Runs all visual and audio inference models in parallel. The "senses" of the system — raw detection and recognition.

Responsibilities: YOLOv8 detection, ByteTrack tracking, logo recognition, PoseFormer action recognition, SlowFast clip inference, Whisper transcription, PANNs audio classification. All models run concurrently on GPU.

INPUTS
FramePackets from Redis AudioChunks from Redis
OUTPUTS
DetectionResults JSON TrackState JSON ActionLogits JSON AudioEvents JSON
FusionAgent
Combines all perception outputs into a unified scene representation. The "understanding" layer — turns raw detections into meaningful scene state.

Responsibilities: Cross-modal transformer fusion over 2s windows, temporal alignment of AV streams, scene embedding generation, event descriptor JSON construction, intermediate storage of fused embeddings.

INPUTS
DetectionResults ActionLogits AudioEvents
OUTPUTS
SceneEmbedding (512-dim) EventDescriptor JSON
ReasoningAgent
Generates high-level scene narratives, anomaly scores, and intent classifications using the LLM. The "brain" — interprets what the scene means.

Responsibilities: Maintains 30s event buffer. Sends structured prompts to Mistral-7B. Parses LLM output into anomaly_score, intent_class, narrative_text. Decides whether to trigger AlertAgent. Writes summaries to DB.

INPUTS
EventDescriptor JSON (30s buffer)
OUTPUTS
SceneNarrative text AnomalyScore [0–1] AlertTrigger event

Agent Communication Flow

StreamAgent → Redis:frames PerceptionAgent → Redis:detections FusionAgent → Redis:events ReasoningAgent → DB + WebSocket Dashboard / Alerts

Decision-Making Hierarchy

All agents are supervised by a lightweight CoordinatorAgent that monitors Redis stream lag, agent health (heartbeats), and GPU memory. If any agent falls behind by >5s, the Coordinator triggers adaptive frame-skip to recover. Priority order: ReasoningAgent > FusionAgent > PerceptionAgent > StreamAgent. The ReasoningAgent's output is final — it can override lower-level alert conditions.

⚠

LLM latency note: ReasoningAgent processes 30s windows with ~3s LLM latency. This means reasoning output lags perception by 30s by design. This is acceptable for context understanding and not a bottleneck — real-time alerts come from PerceptionAgent's rule-based output, not the LLM. Clearly separate these two alert pathways in the dashboard UI.

  • All training scripts in /scripts/train/ using PyTorch Lightning for clean train loops.

IP Camera Integration (RTSP)

# OpenCV RTSP handler with reconnect logic
import cv2, asyncio, time

class RTSPStream:
    def __init__(self, url: str, fps: int = 5):
        self.url = url
        self.fps = fps
        self.cap = None

    async def connect(self):
        self.cap = cv2.VideoCapture(self.url, cv2.CAP_FFMPEG)
        self.cap.set(cv2.CAP_PROP_BUFFERSIZE, 3)
        self.cap.set(cv2.CAP_PROP_FPS, self.fps)

    async def read_frames(self):
        interval = 1.0 / self.fps
        while True:
            ret, frame = self.cap.read()
            if not ret:
                await asyncio.sleep(2)
                await self.connect()  # auto-reconnect
                continue
            yield frame, time.time_ns() // 1_000_000  # timestamp_ms
            await asyncio.sleep(interval)

Edge vs Cloud Architecture

Edge Node (On-Site)

StreamAgent, PerceptionAgent (YOLOv8, ByteTrack, Pose). Runs on local GPU server or Jetson AGX Orin. Processes all camera feeds. Sends only structured JSON events upstream — NOT raw video. Drastically reduces bandwidth.

Cloud / Central Server

FusionAgent, ReasoningAgent (LLM), TimescaleDB, FastAPI, Next.js Dashboard. Can serve multiple edge nodes. Handles heavy LLM inference if edge GPU insufficient. Multi-tenant with per-site isolation.

FYP Demo Mode

All services run on a single GPU workstation via docker-compose. StreamAgent reads local MP4 or RTSP. NVIDIA Container Toolkit enables GPU sharing between containers via MPS.

Scaling to Multiple Cameras

  • Each camera gets an independent StreamAgent instance (separate Docker container, unique camera_id).
  • PerceptionAgent uses consumer groups on Redis Streams — multiple workers pull from the same frame queue in parallel. Add workers = add parallelism, no code change.
  • FusionAgent maintains per-camera state dictionaries. Locks per camera_id prevent cross-camera state corruption.
  • Dashboard uses camera_id filter on all TimescaleDB queries. Each camera has its own event stream and alert channel.
  • For FYP: demonstrate 2–3 simultaneous cameras on a single RTX 3060 (12GB) using CUDA MPS to share GPU across processes.

Model Optimization

TechniqueApplied ToExpected GainTool
TensorRT INT8 YOLOv8, SlowFast 2.5–4× speedup, <2% mAP drop torch2trt / TensorRT Python API. Calibration on 500 representative frames.
ONNX Export ViT logo model, PANNs CNN14 1.5–2× speedup via ONNX Runtime torch.onnx.export(). Use ORT with CUDA execution provider.
GGUF 4-bit Quantization Mistral-7B LLM 8GB → 4GB VRAM. 2× token/s llama.cpp Q4_K_M quantization. Run via Python bindings (llama-cpp-python).
whisper.cpp Whisper medium 3–5× vs PyTorch Whisper C++ inference with cuBLAS CUDA backend. Python binding for subprocess call.
Frame batching PerceptionAgent 40% GPU utilization improvement Collect batch of 4 frames before inference call. Adds ~80ms latency, acceptable for 5fps pipeline.
Adaptive FPS StreamAgent Reduces GPU load 30–50% in idle scenes Drop FPS to 2 when optical flow magnitude is below threshold (scene is static). Auto-ramp back when motion detected.

Latency Reduction Strategy

  • Zero-copy frame passing: Use shared memory (multiprocessing.shared_memory) between StreamAgent and PerceptionAgent. Avoid Redis serialization for frame numpy arrays — only send frame IDs over Redis.
  • Async pipeline: Every agent uses Python asyncio. No blocking I/O anywhere in the hot path. Audio and video preprocessing run concurrently in separate threads.
  • Early exit detection: If frame optical flow is near-zero AND no persons detected in prior 10 frames, skip SlowFast and PoseFormer — save ~80ms per frame.
  • Redis pipeline commands: Batch Redis writes using pipeline context — reduces round-trips from N to 1 per frame cycle.
  • TimescaleDB chunk tuning: Set chunk_time_interval = 1 hour for event tables. Use partial indexes on (camera_id, timestamp) for dashboard range queries.

Risks, Limitations + Mitigations

GPU Memory Overflow
HIGH
Running YOLOv8x + SlowFast + Whisper + Mistral-7B simultaneously may exceed 12GB VRAM. Multiple models competing on shared GPU causes OOM crashes.
Mitigation: Run Mistral-7B on CPU (acceptable at 30s processing cycle). Use model offloading — load SlowFast only when a person is detected (lazy loading). Allocate Whisper to CPU with whisper.cpp. Profile with nvidia-smi during load testing.
RTSP Stream Instability
HIGH
IP cameras drop streams under network stress. OpenCV's VideoCapture silently fails — ret=False without exception. Unrecovered drops stall the entire pipeline.
Mitigation: Implement exponential backoff reconnect logic (1s, 2s, 4s, 8s). Heartbeat check every 10s — if no frame received, trigger reconnect. Dashboard shows camera status (connected / reconnecting / offline).
Real-World Domain Gap
HIGH
Models trained on benchmark datasets (COCO, HMDB51) under ideal conditions perform poorly on actual surveillance footage: low resolution, motion blur, lens distortion, extreme angles, poor lighting.
Mitigation: Domain adaptation fine-tuning on 500 custom IP camera frames. Test-time augmentation (TTA) with horizontal flip averaging. CLAHE preprocessing for low-light. Plan 10% of total project time for domain-specific model tuning.
LLM Hallucination
MEDIUM
Mistral-7B may generate inaccurate scene narratives or incorrect anomaly classifications, especially for ambiguous events. False alert generation undermines system trust.
Mitigation: Constrain LLM output with JSON schema validation. Use few-shot prompting with 5 labeled examples per scene type. Never trigger critical alerts from LLM alone — always require PerceptionAgent confirmation. Display confidence score and "AI-generated" label on all narratives.
AV Synchronization Drift
MEDIUM
Audio and video streams may drift over time (common in RTSP). Misaligned timestamps cause fusion engine to correlate audio events with wrong video frames.
Mitigation: Use PTS (Presentation Timestamp) from FFmpeg — do not rely on wall clock. Implement NTP sync check at stream start. Add drift monitor: if A/V offset >200ms, flag and trigger re-sync. Use FFmpeg's -use_wallclock_as_timestamps 1 for live streams.
Privacy + Ethical Risk
MEDIUM
System processes identifiable human data (faces, movements, speech). Facial recognition, if added, may violate Pakistani data privacy norms. Misuse for unauthorized surveillance.
Mitigation: Do NOT implement facial recognition in Phase 1 — use anonymous tracking IDs only. All stored video footage encrypted at rest (AES-256). Access control: JWT auth required for all API endpoints. Academic use declaration in FYP documentation. No real personal data used in demos.
Limited GPU Hardware
MEDIUM
UCP lab GPUs may have limited availability. Training and running all models on a single consumer GPU is challenging at scale.
Mitigation: Use Google Colab Pro+ (A100) for training. Download pre-trained weights for most models (no training needed). For FYP demo, run single camera at reduced resolution (480p) to demonstrate full pipeline within hardware constraints.
